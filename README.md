# Book Finder Assistant (Telegram)

Telegram-бот для поиска книг по **тексту** и **фото обложки**.

Главная идея: пользователь отправляет описание/цитату или картинку, бот определяет произведение и автора, затем ищет в **Флибусте** (приоритет), а если не уверен — делает fallback в **Google Books**.

## Какие задачи выполняет проект

### 1) Поиск книги по тексту
- Нажми `🔎 Find` в клавиатуре (или используй `/find <запрос>`), затем пришли обычный текст.
- Использует Gemini для извлечения структурированного запроса:
  - `query`
  - `title` (если уверенно)
  - `author` (если уверенно)
  - `confidence`
- Строит несколько попыток поиска и проверяет совпадения по скорингу.
- Приоритетно ищет во Флибусте; если не сработало — обращается к Google Books.

### 2) Поиск книги по фото
- Принимает фото в Telegram.
- Скачивает файл из Telegram и отправляет в Gemini Vision.
- Извлекает вероятные `title/author/isbn/evidence`, а также RU/EN-варианты названия и автора (`title_ru`/`author_ru`/`title_en`/`author_en`/`variants`) для более надёжного сопоставления с русскоязычным каталогом.
- Проверяет совпадение во Флибусте по всем этим вариантам (не только по исходной догадке) и формирует ответ со ссылками скачивания.
- Если уверенного совпадения нет — fallback в Google Books.

### 3) Интеграция с Флибустой
- Поиск книг и авторов через `flibusta-api`.
- Получение метаданных книги (жанры/описание).
- Генерация ссылок на форматы (MOBI/EPUB).
- Нормализация ошибок сети/таймаутов/403/429 в user-friendly сообщения.

### 4) Интеграция с Google Books (fallback)
- Поиск по сформированному запросу.
- Выдача лучшего совпадения и ссылки для открытия книги.
- Используется, когда Флибуста не дала уверенный результат.

### 5) Скоринг и матчинг результатов
- Нормализация текста (регистр, `ё`→`е`, пунктуация).
- Сравнение по exact/partial совпадению title и author.
- Дополнительный token-overlap скоринг для неполных запросов.
- Порог принятия для решения «совпадение/не совпадение».
- Логика вынесена в `src/core/matching.js` и покрыта тестами (`npm test`).

### 6) Кэширование
- Ограниченный (bounded LRU, до 500 записей) кэш ответов: по SHA-256 хэшу изображения для фото и по нормализованному запросу для текстового поиска.
- Уменьшает повторные вычисления и ускоряет ответ на одинаковые запросы, не растёт бесконечно на долгоживущем процессе.

### 7) Режимы отладки
Включаются только через переменные окружения при старте процесса (это **не** Telegram-команды):
- `RAW_MODE=1` — показывает сырой JSON Vision-ответа (владельцу).
- `FLIBUSTA_DEBUG=1` — дебаг запросов к Флибусте (владельцу).
- `GEMINI_DEBUG=1` — дебаг Gemini для текстового извлечения (владельцу).

### 8) UX в Telegram
- Reply-keyboard меню после `/start`:
  - `🔎 Find` — начать текстовый поиск
  - `📩 Kindle email` / `✏️ Изменить Kindle email` — задать/сменить адрес доставки на Kindle
- Регистрирует команды бота через `setMyCommands`.
- Во время долгих операций (Gemini/Флибуста) UI-отклик — это отладочные сообщения владельцу (см. п.7); обычный пользователь ждёт финальный ответ без промежуточного индикатора.

### 9) Kindle-доставка
- Если у пользователя сохранён Kindle email, под найденной книгой во Флибусте появляется кнопка `📩 Send to Kindle`.
- Файл (EPUB, иначе MOBI) скачивается и отправляется вложением на Kindle-адрес.
- Доставка через **Resend** (если задан `RESEND_API_KEY`) или fallback на **SMTP** через `nodemailer`.
- `/smtp_test` (только владелец) — проверяет, что доставка настроена и работает.
- Kindle email обязан заканчиваться на `@kindle.com` или `@free.kindle.com`.

### 10) Контроль доступа пользователей (onboarding)
Проект поддерживает модель доступа с owner-контролем:
- Новые пользователи отправляют запрос доступа кнопкой.
- Владелец получает заявку с кнопками Approve/Reject.
- Есть ручные команды владельца:
  - `/users` — список allowed/pending
  - `/allow <user_id>` (или reply на сообщение)
  - `/deny <user_id>` (или reply на сообщение)
- Есть дневной лимит запросов на пользователя (по умолчанию 15/сутки, владелец не ограничен) — см. `DAILY_LIMIT`.
- Состояние доступа, Kindle email и счётчики лимита хранятся в SQLite (см. «Хранилище данных» ниже).

### 11) Ограничение по топику/треду
- Можно ограничить обработку сообщений конкретным `ALLOWED_THREAD_ID` (0/не задано — без ограничения).

## Стек
- Node.js 20+ (ESM), встроенный `fetch` — отдельный HTTP-клиент не нужен
- Telegraf
- Gemini API (text + vision)
- flibusta-api
- Google Books API
- better-sqlite3 (хранилище: доступы, Kindle email, дневные лимиты)
- Resend / Nodemailer (доставка на Kindle)

> Обрати внимание: `flibusta-api` тянет зависимости, которым для сборки нужен Node **20+** (глобальный `File`); на Node 18 старт упадёт при импорте. Смотри, какую версию Node фактически использует твой деплой.

## Быстрый запуск
1. Установить зависимости:
```bash
npm install
```

2. Создать `.env`:
```env
BOT_TOKEN=...
GEMINI_API_KEY=...
GOOGLE_BOOKS_API_KEY=...
OWNER_ID=...
# optional:
# DATA_DIR=./data
# DB_FILE=./data/bot.sqlite3
# FLIBUSTA_BASE_URL=https://flibusta.is
# DAILY_LIMIT=15
# ALLOWED_THREAD_ID=0
# RAW_MODE=1
# FLIBUSTA_DEBUG=1
# GEMINI_DEBUG=1
# DEBUG_ERRORS=1
# RESEND_API_KEY=...
# RESEND_FROM=onboarding@resend.dev
# SMTP_HOST=... / SMTP_PORT=... / SMTP_USER=... / SMTP_PASS=... / SMTP_FROM=...
#
# Только для миграции со старой версии (не нужны для новых деплоев):
# ACCESS_FILE=./data/access.json
# KINDLE_FILE=./data/kindle.json
# LIMITS_FILE=./data/limits.json
```

3. Запустить:
```bash
npm start
```

4. Прогнать тесты (модуль скоринга/матчинга):
```bash
npm test
```

## Хранилище данных
Доступы, Kindle email и дневные лимиты хранятся в SQLite (`better-sqlite3`, WAL-режим) в файле `bot.sqlite3` внутри `DATA_DIR`.

Если бот раньше работал на JSON-файлах (`access.json`/`kindle.json`/`limits.json`), при первом старте на новом коде эти файлы **автоматически** импортируются в SQLite (без ручных действий), а сами файлы переименовываются в `*.json.bak` (не удаляются — на случай отката). Дальнейшие старты миграцию не повторяют.

## Persistent storage (Railway Volume)
Если деплои сбрасывают данные (доступы/Kindle email/лимиты), подключи Volume и задай:
```
DATA_DIR=/data
```
Тогда файл `bot.sqlite3` будет сохраняться между деплоями.

## Основные команды
- `/start` — старт и меню
- `/find <текст>` — явный текстовый поиск (без аргумента — просит прислать описание следующим сообщением)
- `/users` (owner) — список allowed/pending
- `/allow <user_id>` (owner)
- `/deny <user_id>` (owner)
- `/smtp_test` (owner) — проверка доставки на Kindle (Resend/SMTP)

Режимы отладки (`RAW_MODE`, `FLIBUSTA_DEBUG`, `GEMINI_DEBUG`) — переменные окружения, не команды (см. п.7 выше).
